## 6. Acceptance Verdict

### (a) Why is `dialog_dispatch_locks` empty?

The table is **properly released but the webhook bypasses it entirely**, so the table will typically be empty (or near-empty) in steady state:

1. `compose-dialog-segments` acquires the lock at line 540 with TTL=90s.
2. The lock is **always released** in the `finally` block at line 4084, even on error or early return (202, 422, 409, crash).
3. Typical dispatch latency (preflight ‚Üí Sync.so POST ‚Üí 202 response) is 5‚Äì30s, well within the 90s TTL.
4. Once `compose-dialog-segments` returns, the lock is gone.

The table therefore contains a row only during the brief window between line 540 (acquire) and line 4084 (release). Between dispatches ‚Äî and always inside the webhook ‚Äî it is empty.

### (b) Which dispatch paths bypass the lock entirely?

1. **`sync-so-webhook/index.ts` ‚Äî all branches** (D2, D3, D4, D5): The webhook imports `withDialogLock` but never calls it. Every `UPDATE composer_scenes` in the webhook (COMPLETED pass-done, COMPLETED all-done, COMPLETED NOOP retry, FAILED retry, FAILED scene-fail) is executed **without any lock**. These writes race with concurrent watchdog or client invocations of `compose-dialog-segments`.

2. **`lipsync-watchdog` re-dispatch** (D7): The watchdog itself does not acquire the lock ‚Äî it just invokes `compose-dialog-segments` via `fetch`. If the webhook simultaneously fires `compose-dialog-segments` (D2/D3) AND the watchdog fires it (D7) within 200ms, both race to acquire the lock at line 540. One wins and dispatches; the other gets `202 scene_lock_busy`. The losing invocation disappears silently.

3. **Plan-D `setTimeout` fan-out** (D6): Uses `setTimeout(0ms‚Äì750ms)` which Deno's edge runtime may execute after the parent function returns ‚Äî the parent's lock is **already released** before the fan-out invocations run. Each fan-out acquires its own lock, but there is no coordination between fan-outs of the same scene, so if `concurrencyCap=4` and all 4 simultaneously reach the lock, 3 of them get `202 scene_lock_busy` and are silently dropped.

### (c) Top 3 root causes for scene `cba18767` getting 21 attempts mixing NOOP and `provider_unknown_error`

**Root Cause 1 ‚Äî NOOP ‚Üî FAILED ping-pong on a re-encoded passthrough plate**

The scene's plate is a near-static Hailuo render. `sync-3` with `auto_detect:true` returns COMPLETED but re-encodes without adding mouth motion (byte-similar output). The v127 heuristic (`sizeRatio in [0.65, 1.35]`) fires and recycles the pass as `"retrying"` (R1). The retry (now `coords-pro` via the v126 hard-wire at line 987) hits the same static plate and returns `provider_unknown_error` (FAILED). The FAILED retry (R2) fires another dispatch. That dispatch might again be a sync-3 `auto_detect` preclip job (because preclip_url is cleared at line 999 and re-rendered) ‚Üí COMPLETED NOOP again ‚Üí R1. This 3-way cycle: **NOOP retry ‚Üí FAILED ‚Üí FAILED retry ‚Üí NOOP retry ‚Üí ‚Ä¶** runs until the budgets of BOTH the NOOP guard (`retry_count < 2`) AND the FAILED guard (`passRetryCount < 3`) are independently exhausted. Because the `retry_count` field is shared, once the NOOP guard closes at `retry_count=2`, the FAILED guard sees `passRetryCount=2` and has one more attempt before v121 stop-loss triggers at `passRetryCount >= 1` (confusingly: v121 fires at `passRetryCount >= 1`, meaning only the VERY FIRST retry of a provider_unknown_error without error_code is allowed ‚Äî but only if `forceCoordsRepair` is false; the v30 multi-speaker logic injects `forceCoordsRepair=true` which bypasses v121).

**Root Cause 2 ‚Äî v87 coord-refresh resetting `"done"` passes to `"pending"` on every advance/retry**

Every `compose-dialog-segments` invocation re-runs plate-identity resolution (lines 1128‚Äì1176). If the plate identity result differs slightly between runs (non-deterministic Gemini Vision response), `speakerCoords` change, triggering v87 coord-refresh (line 2100‚Äì2137). At line 2125‚Äì2131, any pass already `"done"` or `"failed"` with changed coords is reset to `"pending"` with `output_url=null, job_id=null`. This means that a scene with 4 speakers that successfully completes passes 0 and 2, but is called again for pass 3's retry, can simultaneously reset passes 0 and 2 to `"pending"` ‚Äî producing phantom re-dispatches (R5) that never complete the scene but do generate Sync.so job attempts.

**Root Cause 3 ‚Äî `syncso_dispatch_log` v118 circuit-breaker counting across scene resets**

The v118 per-pass circuit breaker (lines 2218‚Äì2226) counts ALL `FAILED` rows in `syncso_dispatch_log` for `(scene_id, pass_idx)` with no time bound. When a user resets the scene ("Sauber neu starten"), `dialog_shots` is cleared but `syncso_dispatch_log` is not. The next dispatch rebuilds passes with the same `pass_idx` values (0, 1, 2, 3 for 4 speakers). Old failed rows from the prior run immediately count against the new run's budget of 5. A scene that failed 3 times before reset starts its fresh run with only 2 remaining attempts per pass ‚Äî with 4 speakers √ó 2 remaining attempts = 8 more dispatches, all destined to fail at the circuit-breaker after the cap is hit. Combined with the NOOP/FAILED ping-pong (Root Cause 1) exhausting retries before the circuit-breaker can fire cleanly, scenes accumulate well past 21 total Sync.so job submissions.
