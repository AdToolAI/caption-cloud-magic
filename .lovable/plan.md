
## Plan: Interview-Entwurf wirklich speichern und sauber wieder aufnehmen

### Was ich im Code gefunden habe
Die Speicherung ist aktuell nur halb umgesetzt:

- `UniversalVideoWizard.tsx` speichert zwar `category`, `mode`, `step`, `moodConfig`, `consultationResult`, `autoGeneratedProject`
- `UniversalVideoConsultant.tsx` speichert aber **nur** `messages` und `progress`
- wichtige Zustände fehlen komplett:
  - `lastRecommendation`
  - `showModeChoice`
  - Kategorie-/Modus-Metadaten
  - Zeitstempel / Versionsinfo
- es gibt **keinen echten Entwurf-Dialog**, obwohl genau dieses Verhalten gewünscht ist
- zusätzlich wird der Consultant-State bei erfolgreicher Generierung aktuell wieder gelöscht

Das erklärt sehr gut, warum man wieder bei Phase 1 landen oder einen unvollständigen Restore bekommen kann.

## Umsetzung

### 1. Draft-State vereinheitlichen und robuster machen
**Dateien:**
- `src/components/universal-video-creator/UniversalVideoWizard.tsx`
- `src/components/universal-video-creator/UniversalVideoConsultant.tsx`
- optional neuer Helper z. B. `src/lib/universal-video-draft.ts`

Ich würde die Browser-Speicherung auf ein sauberes Draft-Schema umstellen:

```text
wizard draft:
- category
- moodConfig
- mode
- step
- consultationResult
- autoGeneratedProject
- updatedAt
- version

consultant draft:
- category
- mode
- messages
- progress
- lastRecommendation
- showModeChoice
- updatedAt
- version
```

Ziel:
- nicht mehr zwei lose, unvollständige Snapshots
- Restore nur, wenn Draft gültig und zur aktuellen Videoart passt

### 2. Entwurf-Dialog beim Einstieg einbauen
**Datei:** `src/components/universal-video-creator/UniversalVideoWizard.tsx`

Beim Laden:
- gespeicherten Wizard-Draft **einmal zentral lesen**
- wenn ein sinnvoller Entwurf vorhanden ist (`step > 0` oder Consultant-Nachrichten vorhanden), **nicht sofort hart hydratisieren**
- stattdessen Dialog zeigen:

```text
Entwurf gefunden
Du hast ein begonnenes Video-Interview.
[Fortfahren] [Neu starten]
```

Verhalten:
- **Fortfahren**: gespeicherten Zustand laden
- **Neu starten**: beide Keys löschen und bei Schritt 0 beginnen

Damit passiert genau das gewünschte UX-Verhalten: Entwurf anbieten, nicht ungefragt überschreiben.

### 3. Consultant-Entwurf vollständig speichern
**Datei:** `src/components/universal-video-creator/UniversalVideoConsultant.tsx`

Ich würde die Consultant-Persistenz erweitern, damit nicht nur Chatblasen gespeichert werden, sondern der echte Interview-Fortschritt:

- `lastRecommendation` mitspeichern
- `showModeChoice` mitspeichern
- Kategorie/Modus mitspeichern
- Restore nur, wenn gespeicherter Draft zu `category` und `mode` passt
- `messageIdsRef` weiterhin aus restored messages initialisieren

Wichtig:
Aktuell kann nach Reload zwar evtl. Text da sein, aber der eigentliche Abschlusszustand fehlt. Genau das muss geschlossen werden.

### 4. Kein versehentliches Löschen mehr während normaler Nutzung
**Datei:** `src/components/universal-video-creator/UniversalVideoWizard.tsx`

Ich würde die Löschlogik klar trennen:

- **Nur löschen bei:**
  - `Neues Video starten`
  - bewusstem „Neu starten“ im Draft-Dialog
  - optional „Zurück zur Kategorie“, wenn das wirklich als Neustart gilt

- **Nicht löschen bei:**
  - normalem Zurück-Button
  - Reload
  - Navigation weg und zurück
  - erfolgreicher Generierung, solange der Nutzer nicht explizit neu startet

Konkret würde ich das aktuelle Entfernen von `universal-video-consultant-state` bei `handleAutoGenerationComplete()` überprüfen und sehr wahrscheinlich entfernen, damit Antworten nicht direkt wieder verloren gehen.

### 5. Hydration gegen kaputte oder alte Drafts absichern
**Dateien:**
- `UniversalVideoWizard.tsx`
- `UniversalVideoConsultant.tsx`

Statt mehrfach verteilt `JSON.parse(localStorage.getItem(...))` aufzurufen, würde ich:
- einen zentralen Safe-Parse-Helper nutzen
- Version prüfen
- ungültige / alte Daten verwerfen
- im Fehlerfall sauber auf Neu starten zurückfallen

Das verhindert stille Resets durch defekte oder veraltete Local-Storage-Inhalte.

### 6. Kleine Regression-Absicherung
**Dateien:**
- neue/angepasste Tests unter `tests/` oder komponentennahe Tests

Ich würde mindestens diese Fälle absichern:
- Interview gestartet → Reload → Entwurf wird angeboten
- „Fortfahren“ lädt Phase + Antworten korrekt
- „Neu starten“ leert beide Draft-Keys
- „Neues Video starten“ beginnt wirklich bei 0
- erfolgreicher Render löscht den Entwurf nicht ungewollt

## Betroffene Dateien

## Erwartetes Ergebnis
Nach der Umsetzung soll das Verhalten so sein:

```text
Interview begonnen
-> Seite verlassen / Reload
-> Entwurf-Dialog erscheint
-> Fortfahren lädt exakt den bisherigen Stand

Neues Video starten
-> Entwurf wird gelöscht
-> Flow beginnt wieder bei Schritt 0
```

## Technische Details
Der wichtigste Fix ist nicht nur „mehr localStorage“, sondern:
- vollständigen Interview-Zustand speichern
- Restore zentralisieren
- Draft erst nach Nutzerentscheidung laden
- Draft nicht mehr bei normalen Erfolgs-/Navigationspfaden versehentlich löschen

Die Musik-Problematik ist davon getrennt und sollte separat weiter debuggt werden.
